# Fluff Extermination Policy

Fluff is anything that does not:

- build the system
- verify the system
- define a real boundary
- explain a current claim
- produce reviewer evidence
- document a current architecture decision
- support a current public release

Anything else must be deleted, moved to archive, or removed from authoritative navigation.

## Fluff Categories

- stale version language
- old speculative scores
- unsupported superlative claims
- duplicate roadmaps
- social post drafts
- unused graphics
- challenge language that exceeds evidence
- abandoned feature declarations
- repeated manifesto text
- implied external validation
- implied production authority

## Archive Rule

Exterminate means remove from authoritative current surfaces. If deletion risks losing project history or breaking links, move material to `docs/archive/` and record the move in [ARCHIVE_MANIFEST.md](../archive/ARCHIVE_MANIFEST.md).

If a file is uncertain, remove it from authoritative navigation and record a
specific resolution path in the archive manifest. Do not let unresolved review
buckets suppress claim-scan failures.
