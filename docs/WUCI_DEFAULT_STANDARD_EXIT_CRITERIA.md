# Wuci-Ji Default Standard Exit Criteria

This document defines when Wuci-Ji / Daylight can honestly say it is becoming,
or has become, a default standard. Default-standard status is adoption and
utility, not self-declared supremacy.

## Standard Candidate

Required:

- Formal spec exists.
- Schemas exist.
- Examples exist.
- Negative tests exist.
- Conformance CLI exists.
- Website explains adoption.
- Non-claims are enforced.

## Usable Standard

Required:

- A third-party project can adopt it with one workflow.
- Conformance report is generated automatically.
- Release gates can fail closed.
- Control mapping can be exported.
- Claim scanner rejects unsupported claims.

## Product Standard

Required:

- Versioned release exists.
- Installation path exists.
- CI/CD integration exists.
- Docs are complete.
- Security policy exists.
- Vulnerability disclosure process exists.
- SBOM/provenance exists for Wuci-Ji itself.

## Ecosystem Standard

Required:

- Multiple independent projects use the schema.
- External tools can emit Daylight evidence.
- Conformance reports are comparable.
- Extension registry exists.
- Public governance exists.

## Default Standard

Required:

- Organizations use Daylight as the normal evidence layer for release and
  security claims.
- Security claims without Daylight-style evidence look weaker than claims with
  it.
- Reviewers can ask, "Where is the Daylight evidence?" and receive a
  machine-checkable answer.

## Formal Authority Standard

Required:

- Actual recognized external bodies, auditors, or procurement processes accept
  the evidence format.

This cannot be self-issued.
